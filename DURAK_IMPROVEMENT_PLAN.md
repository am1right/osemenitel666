# Durak Online — План улучшений

Составлен по результатам аудита `durak_game.py` и `durak_routes.py`.
Каждый пункт содержит описание проблемы, что именно нужно сделать и где.

## Блок 1 — Критические баги игровой логики

### #1 · `_check_game_over` сломан для 3+ игроков
**Проблема:** Игра завершается, как только у *любого* игрока заканчиваются карты.
Победителем объявляется тот, кто разыгрался первым. Это неверно — в Дураке
побеждают все, кто избавился от карт, а **дурак** — последний оставшийся с картами.

**Что сделать:**
- Убрать текущую логику "первый без карт = winner".
- Добавить метод `_eliminate_player(pid)` — убирает игрока из `player_ids`,
  фиксирует его место (1-е, 2-е, …), пересчитывает роли.
- `_check_game_over` завершает игру только когда в `player_ids` остался **один** игрок —
  он и есть дурак.
- Добавить поле `self.losers: list[int]` (порядок выбывания) и `self.durak: Optional[int]`.

**Файл:** `api/durak_game.py`

---

### #2 · Нет механизма выбывания игроков
**Проблема:** Связано с #1. Когда игрок заканчивает карты и колода пуста — он должен
выбыть, но продолжает числиться в `player_ids`, получать роли и участвовать в ротации.

**Что сделать:**
- После каждого `_cleanup_round` вызывать `_eliminate_finished_players()`:
  проверять всех игроков у которых 0 карт и колода пуста → вызывать `_eliminate_player`.
- При выбывании пересчитывать `current_attacker` и `current_defender`,
  пропуская выбывших в очереди.
- В `get_full_game_state` добавить поле `eliminated: list[int]`.

**Файл:** `api/durak_game.py`

### #3 · Волновая логика `players_who_threw_this_wave` работает неверно
**Проблема:** Атакующий бросает карту → попадает в `players_who_threw_this_wave` →
не может добросить вторую карту в том же раунде. В реальном Дураке атакующий
может бросать несколько карт подряд (до лимита). Ограничение "один бросок за волну"
должно применяться только к **подкидывающим** (не основному атакующему).

**Что сделать:**
- Разделить логику: `players_who_threw_this_wave` применять только к не-атакующим.
- Основной атакующий (`current_attacker`) может бросать карты многократно,
  пока соблюдаются ограничения по лимиту и рангам.
- Переименовать в `bystanders_who_threw` для ясности.

**Файл:** `api/durak_game.py`

### #7 · Неправильный порядок добора карт
**Проблема:** `draw_for_players` добирает карты всем игрокам одновременно в цикле.
По правилам Дурака порядок добора строгий: сначала атакующий, затем остальные
по кругу, защитник (если отбился) — последний.

**Что сделать:**
- Переписать `draw_for_players(attacker_index, defender_index)` с явным порядком:
  начать с атакующего, идти по кругу, заканчивать защитником.
- Передавать текущие индексы из `_cleanup_round`.

**Файл:** `api/durak_game.py`

### #8 · `get_full_game_state` раскрывает карты всех игроков при `viewer_id=None`
**Проблема:** При `viewer_id=None` в поле `hands` возвращаются реальные карты
всех игроков (список строк). Этот вызов используется в `/start` и результат
уходит в broadcast всем клиентам через WebSocket.

**Что сделать:**
- При `viewer_id=None` всегда возвращать только количество карт (int), не сами карты.
- В `/start` broadcast отправлять `initial_state` без `hands` (только мета-информацию:
  козырь, порядок игроков, у кого ход).
- Каждый клиент должен запрашивать своё состояние через `GET /state?user_id=X`.

**Файлы:** `api/durak_game.py`, `api/durak_routes.py`

### #16 · Двойной broadcast при `game_over` — гонка состояний
**Проблема:** После завершения игры уходят два broadcast подряд: `game_action`
(с полным стейтом) и `game_ended`. Между ними удаляется запись из `active_games`.
При конкуррентных запросах второй ход может обработаться между этими двумя broadcast.

**Что сделать:**
- Удалять из `active_games` **до** отправки broadcast.
- Объединить два broadcast в один: отправлять только `game_ended` с полным финальным стейтом.
- Обернуть весь блок завершения игры в try/finally чтобы гарантировать удаление из `active_games`.

**Файл:** `api/durak_routes.py`

### #17 · Нет таймаута хода
**Проблема:** Игрок может не ходить бесконечно, блокируя всю партию.
Нет серверного таймера, нет авто-пропуска, нет уведомлений.

**Что сделать:**
- Добавить поле `last_action_at: float` (timestamp) в `DurakGame`.
- Фоновая задача (asyncio background task) или проверка при каждом запросе `/state`:
  если с последнего хода прошло > `TURN_TIMEOUT_SEC` (например, 60 секунд) →
  авто-применить дефолтное действие (для защитника — `take_table`,
  для атакующего — `finish_attack`).
- Константа `TURN_TIMEOUT_SEC = 60` в `durak_routes.py`.
- Broadcast `{"type": "turn_timeout", "user_id": pid}` перед авто-ходом.

**Файлы:** `api/durak_game.py`, `api/durak_routes.py`

## ✅ Статус реализации (обновлено)

Все 18 пунктов выше — **сделаны**. Дополнительно закрыты релизные блокеры:
- **Auth:** все ручки Дурака привязаны к проверенному Telegram `initData` (нельзя ходить/платить/смотреть руки за другого).
- **Sweeper:** фоновое авто-завершение брошенных игр + чистка снимков.
- **Ставки:** 5–100000 ⭐, проверка кошелька, при нехватке — редирект в магазин. Банк наполняется ставками при входе; победитель получает банк −5% комиссии; возврат ставки при выходе из лобби ожидания.
- **Дисконнект:** индикатор онлайн/офлайн + авто-форфейт по отвалу (1×1).
